Agent 概念快速入门

2026-06-26
技术 AI
Agent Token Prompt RAG MCP Tool Calling Skill Workflow

两年前，我们惊叹于 ChatGPT 能聊天。今天，AI 已经能自己读文件、调 API、操作电脑、自主拆任务执行——它不是”更会聊了”，而是”进化出了手脚”。

这篇文章带你理清这背后每一个关键概念：从最底层的 Token 和 Context Window，到让 AI 查资料再回答的 RAG，再到统一工具接口的 MCP、沉淀工作流程的 Skill，最后到能自主落地干活的 Agent 和 Harness。

一、基础：AI 怎么”识字”和”记事”

在聊所有高级概念之前，先搞清楚 AI 最底层的两个限制——它们决定了 AI 能”想”多少。

1.1 Token — 文字的最小单元

维度 说明
定义 LLM 处理文本的最小计量与计费单位，不等于”一个单词”
换算 英文约 4 字符 ≈ 1 Token；中文约 1~2 字 ≈ 1 Token
影响 API 费用、速率限制、max_tokens 上限都以 Token 计算
类比 就像你打字时的”每敲一下键盘”，AI 按 Token 数计费

一句话：Token 是 AI 世界的”最小货币单位”——你跟 AI 的每一次交流，最终都变成 Token 在账单上累积。

1.2 Context Window — 单次处理的”记忆上限”

维度 说明
定义 模型单次对话能”看到”的最大 Token 数量（文字+代码+工具输出总和）
容量 200K Token ≈ 15 万中文字 ≈ 一本中等篇幅的书
瓶颈 对话轮次多了、文件读多了，窗口会满——满了就失忆或报错
解决方案 上下文压缩（Compaction）：把早期历史总结成摘要，腾出空间继续工作

Compaction 的工程智慧：Claude Code 实现了 5 层压缩流水线——从零成本的”裁剪无关历史”到”让 LLM 把整个历史总结成结构化 Markdown 快照”，先用便宜的本地操作，实在不行再花 API 钱做重写总结。

关系：Token 是”最小单元”，Context Window 是”单次容量上限”。就像你写文章的字符数 vs 笔记本的页数限制——字太多，本子写不下。

二、指令：怎么让 AI 听懂需求

有了基础能力，下一步是”让 AI 理解你到底要它干什么”。

2.1 Prompt — 指令话术

维度 说明
定义 发给 LLM 的输入文本，告诉它”你是谁、做什么、按什么格式回答”
三层结构 System Prompt（角色定义+全局规则）+ User Prompt（具体任务）+ Assistant 历史回复
本质 Prompt 就是你给 AI 的”工作说明书”

你打开 Claude Code，说”帮我修一下这个 Bug”。这一句话是 Prompt。但背后系统自动拼进去了：你是谁、项目的技术栈、有哪些可用工具、安全规则约束……这些都是 Prompt 的一部分。

2.2 Context Engineering — 信息流管控

维度 说明
定义 有意识地设计 Agent 每一轮能看到什么信息——系统提示、规则、文件、工具输出、记忆
为什么重要 好的上下文工程比换更强的模型更能提升 Agent 表现
公式 上下文工程 = 精选信息 + 组织结构 + 动态裁剪
类比 给 AI 递”精准小抄”，而不是把整本书扔过去

一句话：Prompt 是”一句话指令”，Context Engineering 是”你给 AI 看的全部资料的编排策略”。从 Prompt Engineering 到 Context Engineering，是从”写提示词”到”管控信息流”的范式升级。

三、查证：怎么解决 AI 瞎编

AI 最致命的问题——幻觉（Hallucination）。你问它”公司上个季度销售额是多少”，它不知道，但会编一个数字，而且编得理直气壮。解决方案：让它先查资料再回答。

3.1 RAG — 检索增强生成

维度 说明
定义 在 AI 回答之前，先从外部知识库检索相关文档，作为参考资料塞进 Prompt，再生成答案
三步流程 检索（Retrieval）→ 增强（Augmented，把文档拼进 Prompt）→ 生成（Generation）
解决的问题 LLM 知识过时 + 不知道你的私有资料 → 按文档回答，不瞎编
典型场景 企业知识库问答、智能客服、合同审查

你把公司 200 页的员工手册做成 RAG 知识库。问”年假有多少天”，系统先检索到《员工手册》第 7 章相关段落，拼进 Prompt 变成”根据以下上下文回答：……Question: 年假有多少天？”——回答就基于真实文档，而不是模型瞎猜。

3.2 向量数据库 — RAG 的检索引擎

传统关键词搜索有硬伤：”销售额”搜不到”营收”，”怎么做红烧肉”匹配不到”如何烹饪红烧肉”。

维度 说明
定义 专门存储和检索 Embedding 向量的数据库，支持”语义相似度搜索”
核心原理 把文本转成向量（Embedding）→ 语义相近的内容向量距离近 → 搜”红烧肉”也能找到”红烧排骨”
检索流程 文档切块(Chunking) → 向量化存入 → 用户提问向量化 → 相似度检索 Top-K 片段 → 拼进 Prompt
代表产品 Milvus、PGVector、Chroma、Pinecone

关系：RAG 是”方法论”（让 AI 查资料再回答），向量数据库是”核心引擎”（怎么查）。没有向量数据库，RAG 只能做关键词匹配，效果大打折扣。

四、行动：AI 从”说”到”做”

LLM 本质是文字处理器——只输出文字，不能操作文件、不能调 API、不能上网。让 AI 真正”做事”需要三层能力。

4.1 Tool Calling — 函数调用

维度 说明
定义 LLM 返回结构化 JSON 表达”我要调某某工具，参数是这些”，外层程序执行并反馈结果
本质 模型只负责”做决定”（选哪个工具+传什么参数），程序负责”真执行”
一个循环 用户问”旧金山天气”→ 模型返回 get_weather(location="SF") → 程序调天气 API → 结果还给模型 → 模型用自然语言回答你

这就是 Agent 的”手”——LLM 说”我要读文件”，Agent 帮它读。LLM 说”我要调 API”，Agent 帮它调。LLM 自己什么都没做，但它决定了每一步该做什么。

4.2 MCP — 模型上下文协议（AI 界的 USB-C）

最早的实现方式是把每个工具的集成逻辑硬编码在 Agent 主程序里。这导致了著名的 M×N 集成地狱：N 个 AI 模型 × M 个外部工具 = N×M 套定制集成。5 个模型 × 10 个工具 = 50 套集成。

维度 说明
定义 Anthropic 2024年11月开源的开放标准协议，让 AI 应用通过统一接口连接外部工具和数据源
三个角色 Host（AI 应用，如 Claude Desktop）→ Client（协议组件，1:1 连接 Server）→ Server（把外部系统能力包装成标准化工具）
三种能力 Tools（可执行操作）、Resources（只读数据）、Prompts（提示模板）
行业地位 2025年12月捐赠给 Linux 基金会 Agentic AI Foundation，Anthropic+OpenAI+Block 联合创立，月下载量 9700 万次

就像 USB-C 统一了充电口——MCP 统一了 AI 工具接口。你不需要为每个模型单独写集成代码，只要实现一个 MCP Server，所有支持 MCP 的 AI 应用都能用。

4.3 Computer Use — 像人一样操作电脑

维度 说明
定义 AI 通过虚拟鼠标和键盘，直接操控电脑屏幕上的应用——打开软件、点击按钮、填写表单
与 Tool Calling 的区别 Tool Calling 是”调 API”（结构化、精确），Computer Use 是”看屏幕+操作 GUI”（视觉驱动、通用）
代表产品 Anthropic Claude Computer Use、OpenAI CUA（Computer Using Agent）
典型场景 操作没有 API 的遗留系统、跨多个应用完成复杂流程

五、沉淀：把重复工作流程化

5.1 Skill — 技能包

维度 说明
定义 存放在 SKILL.md 中的 Markdown 指令包，教 Agent 如何完成特定任务
与 Prompt 的区别 Prompt 是”一次性指令”，Skill 是”可复用的流程级指令包”——每次遇到同类任务自动加载
存放位置 项目级 .cursor/skills/ 或个人级 ~/.cursor/skills/
典型场景 PR 审查规范、生成 commit message、处理 PDF 合并、代码审查清单

类比：Prompt 是”每次口头教做菜”，Skill 是”写成菜谱贴在厨房”——下次说”做红烧肉”，Agent 自动按菜谱来，不用从头教。

六、自主：AI 能自己拆任务执行

6.1 Agent — 智能体

维度 说明
核心公式 Agent = Model（智力）+ Harness（手脚+流程）
核心循环 观察 → 规划 → 行动 → 验证 → 重复，直到任务完成
与 LLM 的区别 LLM 是”一问一答的文字处理器”；Agent 是”能自己拆任务、调工具、看结果、再调整的自主系统”

一句话道破本质：Agent 就是所有不需要智能的部分拼在一起，给中间那个真正需要智能的 LLM 打工。 LLM 不会读文件？Agent 帮它读。LLM 不会联网？Agent 帮它搜。LLM 不会执行命令？Agent 帮它跑。

七、落地：控风险、串流程、长期运行

前面的概念让 AI 能”做事”，但要让它在生产环境中可靠运行，还需要三层保障。

7.1 Harness — 执行框架 / 安全外壳

维度 说明
定义 包裹在 LLM 外面的整套软件基础设施，负责 Agent 循环、工具调用、上下文组装、权限审批和用户界面
三大职责 ① 权限控制（7 层纵深防御）② 上下文管理（自动压缩、裁剪、注入）③ 工具调度（并发安全、流式执行）

Harness 的安全哲学：

原则 含义
否决优先（Deny-First） 宁可错误阻止一个安全操作，绝不错误放行一个危险操作
渐进信任（Graduated Trust） 每次新会话，信任从零开始重建——Agent 一开始什么都不能做
纵深防御（Defense in Depth） 7 层独立防线，任何一层都能拦截——从全局禁止列表到高风险显式确认

Harness 是”把裸 LLM 包装成可靠 Agent”的系统工程。Claude Code 的源码中，核心 Agent 循环只占 5%，剩下 95% 全在解决”循环跑起来之后怎么保证不跑偏”。

7.2 Workflow — 工作流

维度 说明
定义 把多步骤任务编排成可重复、有顺序的流程
与 Agent 的区别 Agent 是”自主从零规划每一步”；Workflow 是”步骤和分支相对固定，但每步可调用 LLM/工具”
典型场景 文档更新：提案 → 人工 Review → 按任务清单实施 → 跑 CI → 归档

Workflow 是”标准化流水线”——你知道要做什么，只是用 AI 来加速每一步。Agent 是”自主探索”——你不知道具体怎么走，但告诉 AI 目标，让它自己找路。

7.3 Workspace Agent — 长期数字员工

维度 说明
定义 能持续运行、跨会话保持上下文、拥有持久记忆和工作空间的 Agent
与普通 Agent 的区别 普通 Agent 是”一次性临时工”（会话结束就失忆）；Workspace Agent 是”长期员工”（记住你的偏好、项目上下文、历史决策）
代表 OpenAI Workspace Agents、团队共享智能体
关键能力 持久记忆 + 权限体系 + 团队协作 + 持续学习

八、总结：AI 进化全景图

一句话总结：AI 的进化，就是从”给你一段文字”到”帮你把事办成”的旅程。Token 和 Context Window 是基础限制，Prompt 和 Context Engineering 让 AI 听懂你，RAG 让 AI 不再瞎编，Tool Calling / MCP / Computer Use 给 AI 装上了手脚，Skill 沉淀了重复流程，Agent 让 AI 能自主执行，Harness / Workflow / Workspace Agent 让它能安全、可靠、持续地落地干活。

网络安全业务快速入门

2026-06-24
技术 网络安全
渗透测试 代码审计 漏洞挖掘 红蓝对抗 安全运营 应急响应 威胁情报 网安业务

想入门网络安全，但被各种名词搞晕了？渗透测试、代码审计、漏洞挖掘、红蓝对抗、SOC……它们到底有什么区别？各自具体干什么？彼此之间又是什么关系？

这篇文章帮你理清网安行业的业务全貌，每个业务都配有详细的工作流程拆解。

一、网络安全业务全景图

按视角划分为四大板块：

1. 攻击视角 — 红队技术

业务 简介
渗透测试 模拟黑客攻击，对目标系统进行受控测试，发现并验证安全漏洞
代码审计 审查源代码中的安全缺陷（SQL注入、XSS、反序列化等），白盒视角
漏洞挖掘 研究软件/系统，发现未知漏洞（0day），聚焦前沿攻防
红蓝对抗 渗透测试的升级版，模拟真实APT攻击，实战化演练
安全众测/Bug Bounty 通过众包平台邀请白帽子对产品进行测试，按漏洞付费

2. 防御视角 — 蓝队技术

业务 简介
安全运维/安全运营(SOC) 7×24小时监控日志、告警分析、日常安全巡检
应急响应 安全事件发生后的紧急处置——止损、根除、溯源、恢复
威胁情报 收集、分析攻击者TTPs、IOC等情报，为防御提供预警
日志分析/SIEM/态势感知 大数据+规则引擎，从海量日志中发现异常行为

3. 合规与管理视角

业务 简介
等保测评 网络安全等级保护测评，国家合规要求
风险评估 识别资产→分析威胁→评估脆弱性→计算风险值
安全咨询/方案设计 为企业设计安全架构、制定安全策略

4. 开发安全视角

业务 简介
SDL / 安全开发 在软件开发生命周期的每个阶段嵌入安全活动
DevSecOps 将安全自动化融入CI/CD流水线

按技术方向分为以下领域：

领域 说明
APP安全 Android/iOS 应用逆向、加壳脱壳、协议分析
IoT/车联网安全 智能设备固件分析、通信协议漏洞
云安全 容器安全(K8s)、云原生安全、云上配置审计
工控安全 SCADA、PLC等工业控制系统安全
数据安全 数据分类分级、脱敏、防泄露(DLP)
密码安全 密码算法实现审计、TLS配置、密钥管理
AI安全 对抗样本、模型投毒、提示词注入

二、各业务之间的关系

用一张关联图来说明：

核心关系总结：

  1. 渗透测试 ↔ 代码审计：互补关系。渗透是黑盒（从外攻击），审计是白盒（从内审查），两者结合覆盖更全面
  2. 渗透测试 → 漏洞挖掘：进阶关系。渗透测试做多了会发现工具扫不到的东西，就进入了漏洞挖掘领域
  3. 渗透测试 → 红蓝对抗：升级关系。红蓝对抗比渗透测试范围更广（不限单个系统）、时间更长、对抗更真实
  4. 漏洞挖掘 → 渗透测试/红蓝对抗：支撑关系。挖掘出的0day作为武器库供实战使用
  5. SOC/应急响应/威胁情报：三者是日常运行铁三角——SOC看家、应急响应救火、威胁情报预警
  6. 等保测评：包含了渗透测试+代码审计+风险评估，是合规驱动的综合测评
  7. SDL/DevSecOps：将安全左移到开发阶段，从源头减少漏洞

三、各业务具体工作流程（详细拆解）

3.1 渗透测试流程（PTES标准 — 7大阶段）

渗透测试遵循 PTES（Penetration Testing Execution Standard）技术标准，包括7个核心阶段，通常在 1~4周 内完成：

阶段一：前期交互（Pre-engagement）

目标：明确测试边界，白纸黑字签授权。

子事项 干什么
确定测试范围 哪些IP段、域名、子网、外网/内网？哪些系统、API、APP？
确定测试规则 测试时间窗口（避开业务高峰）、是否允许提权、能否修改/上传数据、能否社工/钓鱼
确定测试类型 黑盒（零知识，模拟外部真实攻击）、白盒（全知识，内部审计视角）、灰盒（部分知识，最常见）
签署授权书 法律保障——这是「合法渗透」和「非法入侵」的分界线
确认联系人 24小时紧急联系方式——一旦搞出故障能立刻沟通

阶段二：信息收集（Information Gathering）

目标：尽可能多地获取目标相关信息，攻击面越大，成功率越高。

（1）被动信息收集（不直接接触目标，利用公开资源）

子项 具体操作 工具/平台
WHOIS查询 查域名注册信息、注册邮箱、DNS服务器 whois命令行、站长工具
DNS信息 查A/CNAME/MX/TXT/NS记录，发现关联域名 dig, nslookup, dnsdumpster.com
子域名挖掘 爆破二级/三级域名，证书透明度(CT)日志查询 Subfinder, OneForAll, crt.sh
搜索引擎侦查 Google Hacking（site/inurl/intitle/filetype），GitHub搜寻泄露的密码/配置/源码 Google Dork, GitHub Search
网络空间测绘 在互联网资产搜索引擎中检索目标暴露面 Shodan, Fofa, ZoomEye, Censys
社工信息收集 领英/脉脉查员工信息（可用于钓鱼）、天眼查/企查查查企业架构 社交媒体、企业信息平台
历史数据 Wayback Machine查看网站历史页面、历史漏洞公告 archive.org, exploit-db
JS/前端信息泄露 从JS文件中提取API端点、隐藏路径、AccessKey等 LinkFinder, JSFinder, 手工审查

（2）主动信息收集（直接与目标交互，会留下痕迹）

子项 具体操作 工具
主机发现 探测哪些IP存活（ICMP/ARP/TCP ACK等多种方式） Nmap -sn, fping
端口扫描 扫描开放端口，识别运行的服务（SYN扫描隐蔽，Connect扫描稳定） Nmap -sS/sT, Masscan（高速）
服务/版本识别 识别端口上运行的具体服务及版本号 Nmap -sV
操作系统识别 通过TCP/IP协议栈指纹判断目标操作系统 Nmap -O, p0f
Web指纹识别 识别CMS（WordPress/DedeCMS等）、Web框架（Spring/Flask等）、中间件（Tomcat/Nginx等） WhatWeb, Wappalyzer, EHole
目录/文件扫描 扫描Web目录和敏感文件（后台入口、备份文件、配置文件、.git泄露等） dirsearch, gobuster, ffuf
CDN识别与绕过 判断目标是否使用CDN，如使用了则通过多地Ping、DNS历史等找到真实IP 多地Ping, DNSDB, 证书查询

常见产出物：资产清单（IP、域名、端口）、网络拓扑草图、Web站点列表、员工邮箱列表

阶段三：威胁建模（Threat Modeling）

目标：基于收集到的信息，分析可能的攻击面，确定”从哪里下手”。

子项 干什么
攻击面分析 哪些端口暴露在外？哪些服务有公开漏洞？哪些Web入口可控？
业务逻辑分析 Web应用有哪些功能模块（登录/注册/上传/支付等），每个功能点可能存在什么安全问题？
权限边界分析 内网/外网边界在哪？DMZ区与核心区的网络隔离情况？
攻击路径规划 画出一到多条攻击链：入口点 → 初始突破 → 提权 → 横向移动 → 抵达核心目标

阶段四：漏洞分析（Vulnerability Analysis）

目标：自动化+手工结合，发现系统存在的安全漏洞。

子项 具体操作 工具
漏洞扫描 全量自动化扫描，快速发现已知漏洞（CVE漏洞、弱口令、配置缺陷等） Nessus, OpenVAS, AWVS
Web专项扫描 针对Web应用的SQL注入、XSS、CSRF、SSRF等OWASP Top 10漏洞 Xray, Nuclei + 模板, Burp Suite主动扫描
弱口令爆破 对SSH/RDP/MySQL/FTP/Web后台等常见服务进行弱口令检测 Hydra, Medusa, SuperWeakPass
手工验证 对扫描结果逐条手工确认——自动化工具误报率高，必须人工验证 Burp Repeater, SQLMap –dbs, 手工构造Payload

手工验证需确认三件事：

  • 判断是否真的存在漏洞（还是只是版本号匹配）
  • 确认漏洞利用条件（需要认证？需要特定参数？）
  • 评估漏洞实际危害（能获取什么级别的权限？）

关键原则：自动化扫描只是”捞鱼”，手工验证才是”确认鱼的大小和种类”。不验证的漏洞报告没有价值。

阶段五：渗透攻击（Exploitation）

目标：利用已验证的漏洞，获取目标系统的初始访问权限（Initial Foothold）。

突破方式 具体技术 常用工具
Web漏洞利用 SQL注入拿数据/写Webshell、文件上传绕过getshell、SSRF打内网、反序列化RCE、模板注入(SSTI) SQLMap, 蚁剑/冰蝎/哥斯拉, ysoserial
系统漏洞利用 利用未修复的CVE（如永恒之蓝MS17-010、BlueKeep等）直接获取系统权限 Metasploit, 公开Exploit-DB脚本
弱口令/默认口令 通过前期爆破或已知默认口令直接登录系统 —
钓鱼攻击 构造钓鱼邮件/页面，诱导员工点击并获取凭据或Shell（通常仅在大型红蓝对抗中使用） GoPhish, Cobalt Strike
近源攻击 Wi-Fi破解、物理接口接入、USB BadUSB等（红蓝对抗场景） Aircrack-ng, Rubber Ducky

获取权限的常见形式：Webshell (命令执行型/文件管理型)、反弹Shell (nc/bash/powershell)、远程桌面/RDP、SSH密钥

阶段六：后渗透利用（Post Exploitation）

目标：从已突破的点向更深处推进，扩大战果，提升权限，最终拿下核心目标。

（1）权限提升（Privilege Escalation）

方向 典型方法 工具
Linux提权 内核漏洞提权(DirtyCow/DirtyPipe)、SUID/SGID可执行文件滥用、sudo配置漏洞、Cron任务劫持、Capability滥用、Docker组提权 LinPEAS, linux-exploit-suggester, GTFOBins
Windows提权 内核漏洞提权、服务权限配置不当（不安全的服务路径/可写服务）、AlwaysInstallElevated、UAC绕过、Token窃取 WinPEAS, Watson, JuicyPotato, PrintSpoofer
数据库提权 MySQL UDF提权/MOF提权、SQL Server xp_cmdshell —

（2）横向移动（Lateral Movement）

技术 说明 工具
凭据窃取 从内存dump密码哈希、浏览器保存密码、配置文件中的明文密码 Mimikatz, LaZagne
哈希传递(PtH) 不破解密码，直接用NTLM哈希认证其他Windows主机 Mimikatz, Impacket
票据攻击 Kerberos黄金票据/白银票据，伪造域控权限 Mimikatz, Rubeus
SSH密钥复用 用已获取的SSH私钥登录其他Linux主机 —
RDP/WMI/PSExec 利用Windows远程管理协议跳板 PsExec, WMIexec

（3）数据窃取与权限维持（测试目标场景）

动作 目的
定位核心数据 搜索数据库连接串、敏感文件、代码仓库、内部文档
数据外传测试 模拟攻击者将数据打包传出（测试DLP和数据防泄露能力）
留持久化后门 计划任务/服务/注册表启动项/SSH公钥/WEB后门（测试后必须全部清除）
痕迹清理 清除Shell历史、日志中操作记录、上传的文件（测试后执行）

阶段七：报告撰写（Reporting）

目标：将测试成果转化为可读、可落地的安全报告。报告质量直接决定测试价值。

报告章节 内容
概述 测试目标、范围、时间、测试类型（黑/白/灰盒）
漏洞清单 每个漏洞的：名称、CVSS评分、风险等级（高/中/低）、发现方式、复现步骤（截图+Payload）、危害说明
攻击链总结 从入口到核心目标的完整攻击路径梳理，让客户直观感受风险
修复建议 按优先级排序：高危→立即修复（如直接RCE、数据泄露）；中危→尽快修复（如信息泄露、弱口令）；低危→择期修复。每项给出具体修复方案而非泛泛的”加强安全配置”
附录 工具清单、Payload列表、时间线

3.2 代码审计流程（6大阶段）

代码审计（Security Code Review）是从源码层面发现安全漏洞的白盒测试方法，通常在 1~4周 完成：

阶段一：前期准备

子项 干什么
获取完整源码 包括项目代码、依赖包(package.json/pom.xml/go.mod等)、数据库建表脚本、配置文件模板
了解技术栈 后端语言（Java/Python/Go/PHP/.NET）、Web框架（Spring Boot/Django/Gin/ThinkPHP）、ORM框架、中间件
了解业务架构 功能模块划分、用户角色与权限体系、外部API调用关系、数据流向
环境搭建 本地部署运行，确保代码可编译、可调试
明确审计范围 全量审计还是仅核心模块？是否包含第三方依赖？

阶段二：自动化扫描

子项 具体操作 工具
SAST扫描 使用静态应用安全测试工具全量扫描，自动检测规则匹配的漏洞模式 SonarQube, Fortify SCA, Checkmarx, Semgrep, CodeQL
SCA扫描 软件成分分析——检测引入的第三方依赖是否存在已知CVE漏洞 OWASP Dependency-Check, Snyk, Trivy
密钥/敏感信息扫描 扫描代码中是否硬编码了密码、API Key、Token等 Gitleaks, TruffleHog
结果初筛 过滤误报（False Positive），标记需要人工深入分析的告警 —

注意：自动化工具仅能发现”模式匹配”类型的漏洞（如使用了危险函数），无法理解业务逻辑。误报率通常在30%~70%。

阶段三：人工审计（核心环节）

这是代码审计最有价值的部分，依赖审计人员的经验和对漏洞模式的理解。

（1）审计方法论：污点追踪（Taint Tracking）

核心思想：追踪外部可控数据（Source）→ 经过的过滤函数（Sanitizer）→ 到达危险操作（Sink）的完整路径。

角色 示例（Java）
Source request.getParameter(), @RequestBody, req.getInputStream(), Cookie值, Header值, 文件内容
Sink stmt.executeQuery(), Runtime.exec(), new FileOutputStream(), ProcessBuilder.start()
Sanitizer ESAPI.encoder(), HtmlUtils.htmlEscape(), 参数化查询(PreparedStatement), 白名单校验

如果 Source → Sink 路径上”没有足够的Sanitizer”，就存在漏洞。

（2）逐类漏洞审计要点

漏洞类型 审计对象 典型危险模式
SQL注入 所有数据库查询语句 拼接SQL字符串（"SELECT * FROM users WHERE id=" + id），未使用参数化查询/ORM
XSS跨站 所有输出到HTML/JS的代码 未转义直接输出用户内容（out.println(userInput)），innerHTML赋值
命令注入 所有调用系统命令的地方 Runtime.exec("ping " + ip) 未过滤参数
文件上传 文件上传功能 仅前端校验、黑名单过滤可绕过、未限制文件类型/大小、未重命名
文件操作 文件读取/写入/下载/删除 路径拼接导致目录穿越（../../../etc/passwd）、任意文件读取/写入
反序列化 ObjectInputStream.readObject(), json解析 不可信数据反序列化，未做类型白名单校验
SSRF 服务端发起HTTP请求的地方 URL参数可控且未限制访问目标（可打内网）
XXE XML解析处 未禁用外部实体解析（DTD/ENTITY）
权限绕过 越权访问 未做身份校验的水平越权（IDOR）、未做角色校验的垂直越权
加密问题 加密/哈希/随机数 硬编码密钥、弱算法(MD5/SHA1/DES/ECB)、不安全的随机数(Random而非SecureRandom)
敏感信息泄露 日志/响应/错误页面 日志打印密码/Token、错误信息泄露堆栈/路径、响应返回多余字段

（3）审计顺序策略

  1. 先高危后低危：先找 SQL注入/命令注入/反序列化/任意文件上传 这种直接 getshell 的
  2. 先外部后内部：先审计对外暴露的接口（Controller/API）再审计内部Service
  3. 先功能点后全局：先审计登录/注册/上传/支付等关键功能，再审计全局过滤器/拦截器
  4. 正向+逆向结合：正向从Controller跟到DAO，逆向从Sink反推到Controller

阶段四：漏洞验证

子项 干什么
搭建测试环境 部署代码到测试环境，确保可正常访问
构造攻击向量 针对审计发现的疑似漏洞，构建具体的Payload进行验证
确认漏洞可利用 验证漏洞是否真实存在、利用条件是否满足、实际危害程度
截图/录屏 记录完整复现过程作为证据

阶段五：修复建议

要求 说明
代码级方案 不能写”修复SQL注入”而应写”将第X行拼接SQL改为PreparedStatement参数化查询”
给出正反对比 错误写法 vs 正确写法，方便开发对照修改
风险评级 按CVSS标准评分，标注危害等级和修复优先级

阶段六：审计报告

与渗透测试报告结构类似，额外包含：漏洞所在文件路径+行号、代码片段截图、修复前后对比代码。

3.3 漏洞挖掘流程（7大阶段）

漏洞挖掘（Vulnerability Research）是主动发现未知漏洞（0day）的研究过程，一条完整的漏洞挖掘链通常需要 数周到数月：

阶段一：选题与定向

子项 干什么
选定目标 选择有研究价值的软件/组件/协议/设备。新上线的产品、大版本更新的流行框架、IoT设备固件是热门方向
信息收集 获取目标版本号、技术栈、历史漏洞(CVE)列表、公开安全公告、补丁对比
确定技术路线 根据目标类型确定：Web应用 → 代码审计+黑盒Fuzzing；二进制程序 → 逆向+Fuzzing；IoT/嵌入式 → 固件提取+协议分析

阶段二：环境搭建

目标类型 搭建方式
Web/网络应用 部署源码或二进制，搭建本地测试环境
二进制程序 安装对应OS版本，配置调试器(WinDbg/gdb)、反汇编器(IDA Pro/Ghidra)
IoT设备 提取固件（binwalk解包）、模拟运行(Firmadyne/QEMU)或实体设备调试(UART/JTAG)

阶段三：攻击面分析

子项 分析内容
输入点枚举 所有接受外部数据的地方：HTTP参数、文件上传、网络协议端口、IPC接口、API参数
协议解析点 自定义协议的解析代码——往往是C/C++编写的，内存安全问题重灾区
文件格式解析 支持的文档/图片/压缩包格式——解析代码复杂、攻击面大
第三方组件 使用了哪些开源库？这些库的哪个版本有已知漏洞或历史安全公告？

阶段四：方法选择与执行

这是漏洞挖掘的核心技术环节，选择哪种方法取决于目标类型和研究方向：

方法 适用场景 具体技术 常用工具
模糊测试(Fuzzing) 任何接受外部输入的程序 生成大量变异/随机输入喂给目标，监控崩溃：黑盒（不需要源码，如AFL）、灰盒（通过插桩获取覆盖率反馈，如AFL++）、协议Fuzzing（Peach, Sulley, Boofuzz）、Web Fuzzing（对API参数进行异常值测试） AFL++, LibFuzzer, honggfuzz, Peach, Burp Intruder
逆向分析 无源码的二进制程序 静态分析：用IDA Pro/Ghidra反汇编后阅读伪代码，寻找危险函数(strcpy/memcpy/gets/scanf)；动态分析：用调试器下断点，跟踪内存操作 IDA Pro, Ghidra, x64dbg, WinDbg, gdb
补丁对比 软件供应商刚发布安全补丁 对比补丁前后的二进制/源码差异 → 定位修复的漏洞位置 → 逆向分析漏洞原理 → 针对未打补丁的版本开发利用 Bindiff, Diaphora, IDA
污点分析 有源码的目标 将漏洞模式抽象为QL查询语句，自动追踪Source→Sink路径 CodeQL, Joern
符号执行 探索深层执行路径 将程序输入符号化，通过约束求解遍历不同执行路径，发现常规测试无法触达的深层漏洞 angr, KLEE, S2E

阶段五：异常捕获与定位

子项 干什么
崩溃监控 Fuzzing时监控目标进程是否崩溃（Segment Fault / Access Violation等），收集崩溃样本
崩溃分类 判断崩溃类型：栈溢出、堆溢出、UAF(Use-After-Free)、double-free、整数溢出等
崩溃去重 多个崩溃样本可能触发同一根因，需要去重（通过崩溃地址/调用栈哈希）
根因定位 利用调试器回放崩溃→定位崩溃点→追踪到漏洞根因（哪里的代码对输入处理不当）
可利用性评估 判断漏洞是否可被利用：可控的EIP/RIP？可控的写操作？信息泄露？

阶段六：PoC与EXP开发

产出物 说明
PoC (Proof of Concept) 证明漏洞存在的验证代码，只需触发崩溃/异常即可。用于提交CVE
EXP (Exploit) 能够实际利用漏洞执行代码的完整利用脚本。包含了ROP链/Shellcode等高级技术

EXP开发涉及的关键技术：

技术 说明
栈溢出利用 覆盖返回地址 → ROP链 → 调用system/execve
堆利用 堆风水(Heap Feng Shui)、Tcache poisoning、Fastbin attack
ASLR绕过 通过信息泄露获取基地址、partial overwrite
NX/DEP绕过 ROP(Return-Oriented Programming)或mprotect修改内存属性
CFG绕过 Control Flow Guard — Windows现代保护机制，需要更精巧的利用链

阶段七：漏洞披露

子项 干什么
联系厂商 通过漏洞奖励计划(Bug Bounty)或直接邮件联系厂商安全团队
CVE申请 向MITRE申请CVE编号，获得全球统一漏洞标识
撰写漏洞报告 详细的漏洞描述、影响版本、复现步骤、修复建议
披露时间线 遵循「负责任的披露」原则——给厂商90天修复期后再公开细节
发表技术文章 在博客/安全会议上分享漏洞挖掘过程和技术心得

3.4 应急响应流程（PDCERF模型 — 6大阶段）

应急响应遵循国际通用的 PDCERF 方法论，强调”分秒必争但步步为营”：

阶段一：准备（Preparation）— 功夫在平时

应急响应做得好不好，70%取决于平时准备的充分程度。

准备工作 具体内容
应急预案 针对勒索病毒、数据泄露、DDoS、Web篡改、钓鱼邮件等常见事件类型，分别制定处置SOP
应急团队 明确角色分工：总指挥、技术分析组(取证/逆向)、业务恢复组、对外沟通组、法务合规组
通讯渠道 建立脱离办公网络（以避免被攻击者监控）的应急通讯群
工具资源 提前备好：取证U盘(Linux Live USB)、抓包工具(Wireshark/tcpdump)、日志分析工具、应急工具箱
基础数据 日常维护：网络拓扑图、资产清单(IP/域名/负责人)、基线配置快照、日常流量基线
演练 定期进行应急演练（桌面推演+实战演练），确保流程不是纸上谈兵

阶段二：检测与研判（Detection & Analysis）

收到告警后的第一件事：判断是不是真实事件，而不是误报。

步骤 做什么
发现渠道 SOC/SIEM告警、IDS/IPS告警、杀软/EDR告警、运维发现异常（CPU跑满/流量突增）、员工上报（收到可疑邮件/电脑变慢）、外部通报（白帽子/监管机构）
初步分析 查看告警来源设备、告警类型、触发时间、关联的其他告警
信息验证 登录受影响主机检查：是否有异常进程？异常网络连接？计划任务？新增用户？最近修改的文件？
判断真伪 关键问题：这个告警是否由正常业务行为触发？这个告警的时间与业务变化是否吻合？是否有多个独立告警相互印证？
定级 根据影响范围和危害程度确定事件级别：Ⅰ级（特别重大） 核心系统被控、大规模数据泄露；Ⅱ级（重大） 重要系统被控；Ⅲ级（较大） 非核心系统异常；Ⅳ级（一般） 低风险告警
启动应急 Ⅰ/Ⅱ级事件立即启动应急响应流程，通知上级和相关部门

阶段三：抑制（Containment）— 先止血

关键原则：先切断攻击链路，再调查取证。犹豫越久，损失越大。

抑制手段 适用场景
网络隔离 将受影响主机断网/VLAN隔离，阻止攻击者进一步操控和横向移动
封禁IP/域名 在防火墙上封禁C2通信地址、恶意IP
下线服务 紧急下线受影响系统（同时切换到备用或维护页面）
重置凭据 立即修改受影响账户和关联账户的密码、重置API Key/Token
阻断进程 Kill恶意进程，停止可疑服务
保留证据 在断网前先做内存dump和网络连接快照——一旦关机或断网，内存中的恶意代码、网络连接状态、加密密钥等关键证据将永久丢失

关键决策：是”先关停再调查”还是”维持运行暗中监控”。前者保安全，后者保溯源证据——需要高层决策。

阶段四：根除（Eradication）— 治本

动作 具体内容
清除恶意文件 删除Webshell、木马、后门程序、恶意脚本
清除恶意账户 删除攻击者创建的系统账户、SSH公钥、数据库用户
清除持久化机制 删除恶意计划任务(crontab/计划任务)、注册表启动项、恶意服务
修补漏洞 修复攻击者利用的漏洞（打补丁、改配置、修复代码）
重置全部凭据 所有可能泄露的密码、密钥、证书全部更换
全网排查 确认攻击者没有在其他系统中留下后门——逐一检查同网段/同AD域的其他主机

阶段五：恢复（Recovery）

动作 具体内容
数据恢复 从干净备份中恢复数据——先确认备份未被感染
系统重建 严重感染的主机应该重装而非修复（你不知道攻击者还改了哪里）
逐步上线 先上非核心系统，监控一段时间无异常后再上核心系统
加强监控 恢复后至少持续高强度监控72小时，关注是否有残余后门触发

阶段六：跟进复盘（Follow-up）

动作 具体内容
编写事件报告 包含：事件时间线（精确到分钟）、攻击路径还原、影响范围、处置过程、根因分析
安全加固 根据根因制定改进计划：修补漏洞、关闭不必要端口、加强监控规则、增加告警覆盖
改进流程 应急响应过程中暴露的问题（沟通不畅？工具不足？授权延迟？）逐一改进
存档与分享 事件报告归档，匿名化后在内部/行业内分享经验教训

3.5 红蓝对抗流程

红蓝对抗是渗透测试的”实战升级版”，模拟真实APT攻击的全流程，通常会持续 2~8周：

与渗透测试的关键区别：

维度 渗透测试 红蓝对抗
目标 尽可能多地发现漏洞 模拟真实攻击，检验检测+响应能力
范围 明确限定目标系统 不设限，一切可达的目标
手段 以技术漏洞利用为主 漏洞+社工+钓鱼+近源+供应链等组合拳
隐蔽性 不要求隐蔽 必须隐蔽，测试蓝队检测能力
持续时间 1~4周 2~8周甚至更长
蓝队是否知情 知道在测试 完全不知情（双盲）
步骤 干什么
1.预攻击准备 C2基础设施搭建（域名/服务器/CDN）、免杀木马制作、钓鱼站点/邮件模板准备
2.外网打点 信息收集+漏洞扫描+利用，获取第一台外网主机权限
3.建立据点 部署C2植入物，建立与C2服务器的加密通信隧道（HTTP/HTTPS/DNS隧道）
4.权限提升 从Web/普通用户权限提升至Administrator/root/SYSTEM
5.横向移动 信息收集（域环境、网络拓扑、敏感资产定位）→ 凭据窃取 → 逐步迁移至核心目标
6.达成目标 获取预设的靶标（域控、核心数据库、CEO邮箱等），证明攻击可达
7.清理痕迹 清除操作日志、删除植入物和工具

3.6 安全运营（SOC）日常工作流程

SOC是组织网络安全的”第一道防线”，7×24小时持续监控：

步骤 干什么 工具/系统
1.日志采集 从全网设备(防火墙/WAF/IDS/服务器/终端/应用)统一收集日志，标准化格式 SIEM (Splunk/ELK/奇安信NGSOC)
2.实时分析 规则引擎+UEBA(用户实体行为分析)+机器学习，实时关联分析海量日志 SIEM规则引擎、AI检测模型
3.告警生成 命中规则或模型异常时产生告警（每天可能有数百到数万条） —
4.事件分级 L1分析师告警分类：误报？低危？高危？真实攻击？ —
5.研判处置 L2分析师深入分析确认为真实威胁的事件，执行处置或升级为应急响应 EDR, 威胁情报平台
6.关闭归档 记录处置过程和结果，沉淀为案例库(KB)，优化检测规则 工单系统

SOC分级运营模型：

级别 人员 职责
L1（一线监控） 初级分析师 7×24轮班看告警，初步分类和过滤误报
L2（事件分析） 中级分析师 深入分析确认真实威胁，执行常见事件处置
L3（高级研判） 高级分析师/安全专家 处理复杂APT攻击、0day利用等高级威胁

3.7 威胁情报工作流程

威胁情报为安全运营和应急响应提供”敌情预警”：

步骤 干什么
1.需求定义 明确需要什么情报：行业威胁态势？特定APT组织动态？最新IOC？漏洞利用情报？
2.情报收集 多源采集：开源情报（安全博客/Twitter/GitHub）、商业情报(Recorded Future/微步在线)、社区共享(ISAC)、内部情报（自身SOC产生的告警数据）
3.情报处理 去重、去伪、标准化格式(STIX/TAXII)、打标签
4.情报分析 关联分析、归因攻击组织、评估情报可信度(高/中/低)和威胁等级
5.情报分发 将IOC推送到SIEM/EDR/防火墙自动阻断；战略情报报告推送给决策层
6.情报反馈 评估情报效用，优化情报需求和采集策略

威胁情报金字塔（从底层到顶层）：

层级 内容 有效期 举例
文件哈希(Hash) 恶意文件MD5/SHA256 小时~天（易变） a1b2c3d4...
IP/域名(IOC) C2服务器地址 天~周 103.xxx.xxx.xxx
网络/主机特征 攻击行为模式 周~月 特定User-Agent/注册表键
工具(Tools) 攻击者使用的工具 月~年 Cobalt Strike, Mimikatz
TTPs（战术技术过程） 攻击行为方法论 年 钓鱼→横向移动→勒索

3.8 等保测评流程

等级保护测评（等保）是国家合规要求，覆盖技术+管理两个维度：

步骤 干什么
1.定级 确定系统安全保护等级（一级~五级，最常见的是二级和三级）
2.备案 向公安机关提交定级报告，完成备案
3.建设整改 根据等级要求进行安全建设（技术+管理），补齐差距
4.测评 测评机构入场，按国家标准进行技术测试（渗透测试、配置核查、漏洞扫描）和管理审核（制度/流程/人员）
5.监督检查 公安机关定期检查，持续合规

为什么新的代码更改叫 Pull Request？

2026-05-20
技术 Git
Git GitHub Pull Request 版本控制 代码协作

在 GitHub、GitLab 等平台上，每当我们要向项目提交代码变更时，操作的对象叫 Pull Request（PR）。 这个名字乍看有点反直觉——明明是我”推送”代码，为什么叫”拉取请求”？

一句话解释

Pull Request 的意思是：我请求你（项目维护者）从我的仓库里”拉取”我刚刚推送的代码变更。

为什么不是 “Push Request”？

我们先用一张图看懂 Git 协作的基本模型：

图中展示了 Fork、Clone、Push、Pull Request、Merge 的基本流向

关键点拆解

角色 动作 权限
普通开发者 Fork + Clone + Push（到自己的远程仓库） 只有自己仓库的写权限
项目维护者 Pull（从你的仓库拉取） + Merge 拥有主仓库的写权限

Pull Request 的本质是一个”请求”：

“我无权直接改主仓库，请你（维护者）执行 git pull 把我的改动合进去。”

如果把”拉取请求”改成 “推送请求（Push Request）”，语义会变成：

“请求获得直接向主仓库推送的权限”

但这并不是代码评审流程的真实情况。

一次完整 Pull Request 的动作拆解

下面这张图展示了从本地修改到 PR 合并的全过程：

分步说明

  1. Fork 主仓库 → 获得自己的远程副本
  2. Clone 到本地 → 创建分支并写代码
  3. Push → 把新分支推送到自己的远程仓库
  4. 打开 GitHub → 点击 New Pull Request
  5. 系统自动比较：
    • 基础分支：主仓库的 main
    • 对比分支：你仓库的 feature-x
  6. 你发出请求 → 维护者看到 “有人请求我拉取他的代码”
  7. 维护者执行合并（git pull 或点击 Merge）

为什么这个命名是合理的？

从 Git 命令角度理解

如果维护者在命令行手动合并一个 PR，他会这样做：

你看，动作正是 pull——拉取别人的远程分支到自己的本地仓库。

因此，Pull Request 准确描述了维护者将要执行的操作，而不是贡献者的动作。

POST请求 -> 参数在请求体中
