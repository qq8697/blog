2024 年，Agent 开发是这样的：一个巨大的 System Prompt，几百行，塞满了 `You are an expert...`、`If...then...`。改一个字，整个 Agent 就跑偏了。

2026 年，Agent 开发走向了**解耦**：Prompt 被拆成 Skills（怎么做）和 Rules（不能做）；接口被标准化为 MCP；编排分化出两条路径——写代码深度定制，或拖拽配置快速上线。

**路径 A：代码/框架驱动（工程化）**——用 LangGraph / CrewAI / AutoGen 写代码，深度定制复杂业务逻辑，适合生产级部署。
**路径 B：配置/平台驱动（产品化）**——用 Dify / Coze / Flowise 可视化编排，Skills + MCP + Rules 组合配置，适合快速原型和标准化场景。

两条路径共享同一套底层公式：

> **Agent 开发 = 能力定义（Skills/MCP）+ 行为约束（Rules）+ 推理引擎（LLM）+ 编排载体（框架/平台/代码）**

这篇文章按公式逐层拆解：每一层是什么、两条路径各怎么做、2026 年主流工具怎么选。

---

## 一、核心公式与两条路径

### 1.1 统一公式拆解

```
Agent 开发 = 能力定义 + 行为约束 + 推理引擎 + 编排载体
              │            │          │          │
           Skills/MCP    Rules     LLM 选型   框架/平台
              │            │          │          │
         "能做什么"    "不能做什么"  "怎么思考"  "怎么执行"
```

这四个组件缺一不可：能力定义决定了 Agent 的操作范围，行为约束划定了安全边界，推理引擎提供智力支撑，编排载体负责把一切串起来跑。

### 1.2 路径 A vs 路径 B

| 维度 | 路径 A：代码/框架驱动 | 路径 B：配置/平台驱动 |
|------|----------------------|----------------------|
| 开发方式 | 选框架 → 写代码（LangGraph/CrewAI/AutoGen） | 组合 Skills + MCP + Rules（Dify/Coze/Flowise） |
| 能力定义 | 框架内置 Tool + 可选 MCP | 统一 MCP 协议，即插即用 |
| 行为约束 | 代码中实现安全检查 | 平台内置 Rules + Hooks |
| 推理引擎 | 自由切换模型，代码控制 | 平台内置模型路由 |
| 编排载体 | 代码定义 Agent Loop + 多 Agent 协作 | 可视化 Workflow + 标准化编排模式 |
| 适合人群 | 开发者、工程师 | 开发者 + 业务人员 + 产品经理 |
| 核心优势 | 深度定制、复杂逻辑、完全掌控 | 快速原型、低门槛、开箱即用 |

**一句话总结**：路径 A 是"造轮子"——完全掌控，适合深度定制；路径 B 是"拼乐高"——快速搭建，适合标准化场景。两条路径在 2026 年都已成熟，选哪条取决于你的场景，而不是你的"段位"。

路径 A 的核心框架生态数据：**LangGraph** GitHub 超 9 万 Star，是生产级 Agent 工作流的事实标准；**CrewAI** 月下载量达 520 万。这些不是玩具——大量团队已经在生产环境中使用它们。

### 1.3 统一架构全景图

```
┌──────────────────────────────────────────────────────────────┐
│                     🧠 推理引擎（LLM）                         │
│   Claude 5 / GPT-5.1 / DeepSeek / Qwen 3 / Gemini 3         │
│   混合路由：深度推理模型做 Planning · 轻量模型做 Action         │
├──────────────────────────────────────────────────────────────┤
│                     🔧 能力定义层                             │
│   Skills（Markdown 技能包） ←→  MCP Server（标准化工具接口）    │
│   "能做什么"                                                │
├──────────────────────────────────────────────────────────────┤
│                     🛡️ 行为约束层                             │
│   Rules（项目约定） + Hooks（自动检查）                         │
│   "不能做什么" + "自动检查"                                   │
├──────────────────────────────────────────────────────────────┤
│                     ⚙️ 编排载体                               │
│   路径 A：LangGraph / CrewAI / AutoGen（代码驱动）             │
│   路径 B：Dify / Coze / Flowise（可视化配置）                  │
│   "怎么执行"                                                │
├──────────────────────────────────────────────────────────────┤
│                     🏗️ Harness（生产就绪）                    │
│   权限控制 · 上下文管理 · 错误恢复 · 成本控制 · 可观测性 · 人工介入 │
│   "怎么不跑偏"                                              │
└──────────────────────────────────────────────────────────────┘
```

这张图是全文的导航地图：自上而下，从"大脑"到"手脚"到"安全网"，每一层都会在后续章节展开。

## 二、能力定义：Skills 与 MCP

**核心观点**：能力定义层决定了 Agent 的"手"能伸多远——Skills 定义它懂什么流程，MCP 定义它能接什么系统。

### 2.1 Skills：Markdown 即技能包

Skills 是存放在 `SKILL.md` 中的 Markdown 指令包，教 Agent 如何完成特定任务。它把"每次手写 Prompt"变成了"菜谱式复用"。Skills 的本质是让传统的 System Prompt 走向结构化、模块化——在 IDE 中它表现为 `.cursor/rules/` 下的配置文件或 `AGENTS.md`，在平台上表现为封装好的技能包。

| 维度 | 说明 |
|------|------|
| 存放位置 | 路径 A：项目级 `.cursor/skills/` 或 `.claude/skills/`；路径 B：平台内置 Skill 市场 |
| 触发方式 | 自动匹配（根据用户意图）或手动 `/skill-name` |
| 生态现状 | 2026 年已有数千个社区 Skills，覆盖 TDD、代码审查、文档生成、部署等 |
| 路径差异 | 路径 A 中 Skills 增强代码可控性；路径 B 中 Skills 是开发的核心载体 |

**示例：一个 TDD Skill**

```markdown
# Skill: TDD 开发

## 触发条件
当用户要求"用 TDD 方式开发"或"先写测试再写代码"时激活。

## 工作流程
1. 分析需求 → 列出测试用例
2. 先写测试代码（Red）
3. 运行测试，确认失败
4. 写最小实现代码（Green）
5. 运行测试，确认通过
6. 重构优化（Refactor）
7. 循环直到所有测试通过

## 输出格式
- 测试文件：`__tests__/xxx.test.ts`
- 实现文件：`src/xxx.ts`
```

### 2.2 MCP：AI 界的 USB-C

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，让 AI 应用通过统一接口连接外部工具和数据源。一个 MCP Server 写一次，所有支持 MCP 的 AI 应用都能用——解决了 M×N 集成地狱。

| 维度 | 2026 年现状 |
|------|------------|
| 生态规模 | 2000+ 公共 MCP Server，覆盖数据库、云 API、监控、CI/CD |
| 平台支持 | GitHub、Slack、Jira、Sentry 等主流平台提供官方 MCP |
| 协议演进 | 支持 HTTP streamable transport，不再局限于 stdio |
| 行业地位 | Anthropic + OpenAI + Google + Microsoft 全部支持 |

### 2.3 Computer Use：无 API 系统的最后拼图

Tool Calling 解决了"有 API 的系统"，MCP 统一了接口标准，Computer Use 解决了"没有 API 的遗留系统"——AI 直接看屏幕、点鼠标、敲键盘。目前适用场景有限，但填补了关键空白。

### 2.4 工具选型决策树

```
需要连接外部系统？
  ├── 有 API → 写 MCP Server（一次编写，处处可用）
  ├── 无 API → 考虑 Computer Use（GUI 操作）
  └── 简单操作 → 直接用 Tool Function（函数调用）
```

## 三、行为约束：Rules 与 Hooks

**核心观点**：能力定义让 Agent "能做事"，行为约束让 Agent "不乱来"——这是从 Demo 到生产最关键的一步。

### 3.1 Rules：持久化的项目约定

Rules 是每次对话自动注入的项目级指令，约束 Agent 的行为边界。路径 A 放在 `.cursor/rules/` 或项目根目录的 `AGENTS.md` 中；路径 B 在平台设置中配置。

```markdown
# .cursor/rules/coding-style.md
- 所有 API 必须写 JSDoc
- 测试用 Vitest，覆盖率 ≥ 80%
- 分支命名：feature/xxx, fix/xxx
- 禁止 force push 到 main
- 数据库用 PostgreSQL，不用 MySQL
```

### 3.2 Hooks：自动化安全网

Hooks 在 Agent 执行工具前后触发，用于校验、审计或自动化拦截。路径 A 通过配置文件定义，路径 B 通常内置在平台中。

```json
{
  "preToolUse": [
    {"matcher": "Bash", "hooks": ["./hooks/validate-command.sh"]}
  ],
  "postToolUse": [
    {"matcher": "Write", "hooks": ["./hooks/run-linter.sh"]}
  ]
}
```

### 3.3 三者协同

```
Skills  = 告诉 Agent "怎么做"（流程级指令）
Rules   = 告诉 Agent "什么不能做"（约束边界）
Hooks   = 在 Agent 行动时"自动检查"（安全网）

三者配合使用，而非各自独立：**Skills 定义正向流程，Rules 划定负向边界，Hooks 在边界上执行检查**。
```

**为什么这很重要？** 因为 Skills + Rules + Hooks 解决了 Agent 开发中最核心的痛点：**可复用性**和**可预测性**。你不再需要每次对话都重复解释"我们怎么做 TDD"，你只需要写一次 Skill，永久生效。

## 四、推理引擎：模型选型与上下文工程

**核心观点**：2026 年的模型已经足够强，瓶颈不在模型能力，而在"喂给模型的信息质量"。

### 4.1 2026 模型格局

| 模型 | 定位 | 适合场景 |
|------|------|---------|
| Claude 5 | 深度推理、长上下文（200K） | 复杂代码、长文档分析 |
| GPT-5.1 | 通用能力强、生态最完善 | 全场景覆盖 |
| DeepSeek | 性价比极高、中文优秀 | 中文场景、成本敏感 |
| Qwen 3 | 阿里系、多模态 | 企业级中文应用 |
| Gemini 3 | 多模态、Google 生态 | 多媒体处理 |

**模型选型建议**：路径 A 可以自由组合模型——2026 年流行的做法是**混合路由架构**：用具有深度推理能力的模型（如 Claude 5、GPT-5.1）做 Planning（规划），用更轻量、响应更快的模型做 Action（工具调用执行）。路径 B 通常由平台内置模型路由自动选择。

### 4.2 上下文工程三原则

上下文工程（Context Engineering）是 2026 年 Agent 开发的核心技能，比 Prompt Engineering 更关键：

1. **精选 > 堆量**：100 行精准上下文 > 5000 行无关信息。给 Agent 递"精准小抄"，不是"整本百科全书"
2. **分层注入**：System Prompt → Skills → Rules → 对话历史 → 工具输出，按优先级分层
3. **动态裁剪**：根据当前任务自动选择注入哪些 Skills，不是全部加载

### 4.3 路径 A vs 路径 B 的上下文策略

| 维度 | 路径 A | 路径 B |
|------|--------|--------|
| 上下文控制 | 代码精确控制每一层上下文 | 平台自动管理，手动微调 |
| 记忆管理 | 自建 Memory 层（向量库/摘要） | 平台内置记忆管理 |
| 优化空间 | 极大，可深度定制 | 受限于平台能力 |

## 五、编排模式：从单 Agent 到多 Agent

**核心观点**：能力定义、行为约束、推理引擎都是"零件"，编排载体是把零件组装成"机器"的方式。

### 5.1 Agent Loop 的本质

无论是路径 A 还是路径 B，Agent 的核心运行逻辑都是同一个循环：

```
观察 → 规划 → 行动 → 验证 → 观察 → ...
```

这个循环被称为 ReAct（Reasoning + Acting）模式。路径 A 用代码实现这个循环，路径 B 由平台内置引擎驱动。

### 5.2 四种多 Agent 模式

| 模式 | 结构 | 适用场景 | 成熟度 | 路径 A 实现 | 路径 B 实现 |
|------|------|---------|--------|-----------|-----------|
| Supervisor | 一个主管分配任务给多个 Worker | 代码审查 + 修复 | ✅ 成熟 | CrewAI / AutoGen | 平台内置 |
| Swarm | 多个平等 Agent 自主认领任务 | 并行搜索 + 汇总 | ✅ 成熟 | 代码编排 | 平台内置 |
| Handoff | Agent A 完成后交接给 Agent B | 客服 → 技术专家 | ✅ 成熟 | 代码编排 | 平台内置 |
| DAG | 固定流程 A→B→C | 文档处理流水线 | ✅ 成熟 | 代码编排（LangGraph） | 可视化拖拽 |

**四种模式拓扑图**：

```
Supervisor（主管模式）          Swarm（蜂群模式）
    ┌─────┐                    ┌───┐ ┌───┐ ┌───┐
    │主管  │                    │ A │ │ B │ │ C │
    └──┬──┘                    └─┬─┘ └─┬─┘ └─┬─┘
  ┌───┼───┐                    └──┬───┴──┬──┘
  ▼   ▼   ▼                       ▼      ▼
┌──┐┌──┐┌──┐                  任务池   结果汇总
│W1││W2││W3│
└──┘└──┘└──┘

Handoff（交接模式）             DAG（有向无环图）
┌───┐    ┌───┐                ┌───┐
│ A ├───→│ B │                │ A │
└───┘    └───┘                └─┬─┘
                                │
                           ┌────┼────┐
                           ▼    ▼    ▼
                         ┌──┐┌──┐┌──┐
                         │B ││C ││D │
                         └─┬┘└──┘└─┬┘
                           │        │
                           └───┬────┘
                               ▼
                             ┌──┐
                             │E │
                             └──┘
```

### 5.3 Subagent：当前最佳实践

主 Agent 把特定任务委派给 Subagent，每个 Subagent 拥有独立上下文窗口。好处是：
- 避免上下文污染（主对话不会被大量搜索结果塞满）
- 并行执行（多个 Subagent 同时工作）
- 失败隔离（一个 Subagent 崩溃不影响其他）

### 5.4 路径 A vs 路径 B 编排差异

| 维度 | 路径 A | 路径 B |
|------|--------|--------|
| 编排方式 | 写代码定义角色和流程 | 可视化拖拽 + 配置 |
| 灵活性 | 极高，可任意定制 | 受限于平台能力 |
| 调试 | 传统代码调试工具 | 平台内置 Trace |
| 复用 | 代码库 | 模板市场 |

## 六、Harness Engineering：生产就绪的七个维度

**核心观点**：Agent 循环本身只占 5% 的代码，剩下 95% 全在解决"循环跑起来之后怎么保证不跑偏"。这就是 Harness Engineering 的价值。

行业数据显示，大约 **80%-90% 的 AI Agent 项目没能进入生产环境**——不是因为 AI 不够聪明，而是因为没有人搭好它"干活的环境"。Harness Engineering 就是解决这个问题的。

| 领域 | 核心问题 | 2026 年最佳实践 | 路径 A 做法 | 路径 B 做法 |
|------|---------|---------------|-----------|-----------|
| 权限控制 | Agent 会不会 rm -rf？ | 7 层纵深防御 + Deny-First | 自建权限层 | 平台内置 |
| 上下文管理 | 对话长了就失忆？ | 5 层压缩流水线 + 自动总结 | 自建 Memory | 平台内置 |
| 工具调度 | 并发调用安全吗？ | 并发感知执行器，互斥串行 | 代码控制 | 平台调度 |
| 错误恢复 | 工具调用失败怎么办？ | 自动重试 + 自我修正 + 熔断 | 自建重试逻辑 | 平台内置 |
| 成本控制 | 一次对话花了多少钱？ | Token 预算 + 实时追踪 + 告警 | 自建监控 | 平台仪表盘 |
| 可观测性 | Agent 内部发生了什么？ | OpenTelemetry + 调用链追踪 | 自建 Trace | 平台内置 |
| 人工介入 | 关键操作谁确认？ | Human-in-the-loop + 审批策略 | 自建审批流 | 平台内置 |

> **黄金法则**：永远不要让 Agent 在没有人类确认的情况下，直接向客户发送邮件或操作真实钱包。无论是路径 A 的代码 `interrupt` 机制，还是路径 B 平台的审批节点，**人类作为最后一道防线**是 2026 年商业化落地的硬性要求。

**关键认知**：路径 A 在 Harness 上投入更大，但换来完全掌控；路径 B 开箱即用，但定制空间有限。这是两条路径最核心的 trade-off。

## 七、IDE 与开发工具链

**核心观点**：2026 年的 IDE 不再是代码编辑器，而是 Agent 的控制面板。三条路径互补使用。

| 范式 | 代表 | 特点 | 适合谁 |
|------|------|------|--------|
| 智能补全 | GitHub Copilot、Tabnine | 行级补全，速度快 | 习惯传统编码的开发者 |
| Agent IDE | Cursor、Windsurf | AI 作为伙伴，多文件修改，可视化 Diff | 全栈开发者 |
| 终端 Agent | Claude Code、Codex CLI | 无 IDE 锁定，终端原生，全系统访问 | 偏好终端的高级开发者 |

**关键认知**：开发者不需要三选一——它们是互补的：
- 日常编码 → Copilot 行级补全（零延迟）
- 重构/新功能 → Cursor Agent 模式（可视化 Diff）
- CI/CD/批量任务 → Claude Code CLI（脚本化运行）

**路径 A 工具链**：Cursor/Claude Code + LangGraph/CrewAI + 自建 MCP + Git
**路径 B 工具链**：Dify/Coze 平台 + 平台内置 MCP 市场 + 模板市场

## 八、场景化选型矩阵

| 场景 | 推荐技术栈 | 路径 | 为什么 |
|------|-----------|------|--------|
| 快速原型，业务验证 | Dify / Coze / Flowise | 路径 B | 拖拽式编排，小时级出 Demo |
| 非开发者参与（产品/运营） | Dify / Coze 低代码平台 | 路径 B | 无需编程，业务人员直接配置 |
| 标准化客服/助手 | Dify + Skills + MCP + Rules | 路径 B | 场景固定，配置化足够 |
| 个人开发者，日常提效 | Cursor + Claude + Skills + 社区 MCP | 路径 A | 轻量级代码方案，开箱即用 |
| 团队协作，统一规范 | Claude Code + Rules + Hooks + 自定义 MCP | 路径 A | 版本控制 Skills，统一行为边界 |
| 多 Agent 复杂协作 | CrewAI / AutoGen + Supervisor 模式 | 路径 A | 代码定义角色，深度控制编排 |
| 企业级，安全合规 | AutoGen / 自建 Harness + 7 层权限 + 审计日志 | 路径 A | 安全第一，合规优先，完全掌控 |
| CI/CD 集成 | Claude Code CLI + GitHub Actions + MCP | 路径 A | 脚本化运行，全自动 |

### 路径选择决策树

如果你还不确定选哪条路，按这个流程"对号入座"：

```
开始
  ├─ 你有编程基础吗？
  │   ├─ 否 → 路径 B（Dify/Coze）
  │   └─ 是 ↓
  ├─ 需要深度定制/复杂编排吗？
  │   ├─ 否 → 路径 B（配置化足够）
  │   └─ 是 ↓
  ├─ 需要生产级部署/安全合规吗？
  │   ├─ 是 → 路径 A（LangGraph/AutoGen + Harness）
  │   └─ 否 → 路径 B（快速验证，后续可迁移）
  └─ 最终选择
```

> **提醒**：路径不是"终身绑定"。很多团队的实际做法是：先用路径 B 快速验证想法，确认可行后，核心模块迁移到路径 A 做深度定制。两条路径可以共存。

---

## 九、双路径学习路线

### 路径 A：代码/框架驱动（适合有编程基础的开发者）

```
第 1 周：选一个框架，跑通第一个 Agent
        → LangGraph 或 CrewAI，理解 Agent Loop、Tool Calling、Memory
        🎯 产出：一个能回答公司内部文档问题的 Agent，跑通完整 Agent Loop

第 2 周：写一个 MCP Server
        → 让 Agent 调用你的 API，理解标准化接口的价值
        🎯 产出：一个可复用的 MCP Server，Agent 能通过它查询数据库

第 3 周：用框架定义多 Agent 协作
        → 用 CrewAI 定义 3-5 个角色协作完成任务（如自动写周报）
        🎯 产出：一个多 Agent 协作 Demo，自动完成端到端任务

第 4 周：配置 Rules + Hooks，建立安全边界
        → 让 Agent 在可控范围内工作，权限控制、错误恢复
        🎯 产出：项目级 Rules 配置 + Hooks 脚本，Agent 行为受控

第 5 周：学习上下文工程
        → 优化 Agent 表现，精选信息 > 堆量信息
        🎯 产出：优化后的 Agent，在相同任务上表现明显提升

第 6 周：部署 + 监控 + 成本控制
        → 从本地 Demo 到生产环境，关注 Harness Engineering
        🎯 产出：一个可对外演示的生产级 Agent 原型
```

### 路径 B：配置/平台驱动（适合快速上手，无需深厚编程基础）

```
第 1 周：用 Dify / Coze 搭一个客服 Agent
        → 拖拽式编排，感受"配置驱动"的威力
        🎯 产出：一个能处理常见 FAQ 的客服机器人，在平台上线可对外演示

第 2 周：学习 Skills 写法
        → 理解 Skills 怎么组织、怎么复用、怎么触发
        🎯 产出：3-5 个可复用的 Skill，覆盖常用业务场景

第 3 周：配置 Rules + Hooks
        → 建立安全边界，让 Agent 在可控范围内工作
        🎯 产出：完整的 Rules 配置，Agent 不会越权操作

第 4 周：接入 MCP，扩展 Agent 能力
        → 连接数据库、API、第三方服务
        🎯 产出：Agent 能通过 MCP 访问外部系统，能力范围大幅扩展

第 5 周：学习上下文工程
        → 优化 Agent 表现，精选信息 > 堆量信息
        🎯 产出：优化后的 Agent，回答准确率明显提升

第 6 周：平台部署 + 监控 + 成本控制
        → 从平台 Demo 到生产环境，关注 Harness Engineering
        🎯 产出：一个在生产环境稳定运行的 Agent 应用
```

> **选哪条？** 如果你有编程基础且需要深度定制，走路径 A；如果你想快速验证想法或让业务团队参与，走路径 B。两条路径的底层知识（MCP、Skills、Rules、上下文工程、Harness）是相通的，学会一条再学另一条，成本很低。

> **小结**：2026 年的 Agent 开发，两条路径并存，核心公式统一：**能力定义（Skills/MCP）+ 行为约束（Rules）+ 推理引擎（LLM）+ 编排载体（框架/平台/代码）**。路径 A 用代码掌控编排，路径 B 用配置加速落地。无论选哪条路，你都不再只是"程序员"，而是"Agent 架构师"：设计 Agent 能做什么、不能做什么、怎么协作、怎么监控。
